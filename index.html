System-Design One-Pager (print this)

Trigger mantra: SPEC ‚Üí NUMBERS ‚Üí FLOW ‚Üí FAILURE ‚Üí TRADE-OFFS ‚Üí ROADMAP.

90-second opening script (memorise):

  1. ‚ÄúGoal: X delivers Y to Z users. Non-goals: A/B. Success = p95 latency ‚â§ L, freshness ‚â§ F, cost ‚â§ C.‚Äù
  2. ‚ÄúScale assumptions: QPS, payload size S, daily writes/reads, growth G.‚Äù
  3. ‚ÄúCritical path: request ‚Üí validate ‚Üí write/read ‚Üí persist ‚Üí respond (happy path first).‚Äù
  4. ‚ÄúState model + data model at a glance (entities, keys, hot partitions).‚Äù
  5. ‚ÄúWe‚Äôll design for 1 failure now (idempotency + retries), and evolve here (roadmap).‚Äù

Back-of-envelope (fill in fast):

  • Throughput: req/s √ó payload ‚Üí network/CPU budget.
  • Storage: daily writes √ó avg size √ó retention (+ replication factor).
  • Latency budget: front-end ‚â§50ms, edge ‚â§20ms, core read/write ‚â§120ms, queue/async ‚â§1‚Äì5s (MVP).
  • Costs: biggest 2 line items (storage + egress/compute); name them.

Data model checklist:

  • Entities & keys (avoid hot keys; include secondary indexes).
  • Partitioning/sharding rule (what‚Äôs the shard key and why).
  • State machine for long-running work (e.g., PENDING ‚Üí RUNNING ‚Üí RETRYING ‚Üí SUCCEEDED|FAILED(DLQ)).

Data-flow narrative (say out loud, in order):

  • Ingress format & size ‚Üí validation ‚Üí write path ‚Üí derived data (indexes/vectors) ‚Üí read path & ranking.
  • Idempotency key + retry policy on every write/async step.
  • Exactly which store each read hits (and why).

Failure inventory (pick 2 now, note the rest):

  • Duplicate requests; partial writes; queue poison message; downstream timeout; hot partition.
  • Controls: idempotency keys, exponential backoff, DLQ with alert, circuit breaker, TTLs, SLO + error budget.

Trade-off playbook (2 pros/2 cons, commit):

  • SQL vs NoSQL; stream vs batch; cache vs no cache; single region vs multi.
  • Name what you give up and how you‚Äôd reverse the decision later.

Roadmap ladder (close strong):

  • MVP (90 days): 1 region, 1 DB, 1 queue, observability baseline.
  • Next 10√ó: read replicas/ANN index, cache for N hot queries, backfill job, multi-AZ/region, privacy/PII pipeline.

Security & compliance in one breath: AuthN (JWT) ‚Üí AuthZ (tenant/resource checks) ‚Üí data segregation ‚Üí PII handling (hash/tokenise, deletion path) ‚Üí audit logs.

Why this beats your Synthesis attempt

  • You start with the critical path and state machine, not gateways/caches/ES.
  • You limit primitives (one DB + one queue) until reads prove otherwise.
  • Every box exposes input, output, SLO, and failure behaviour‚Äîno black boxes.
  • You anchor with numbers so choices feel intentional, not ornamental.

Deep keys (mental models you can reuse)

  • Two-Store Rule (MVP): one blob store + one OLTP/OLAP (or OLTP+pgvector). Add ES/Redis only when a measured query needs it.
  • One-Queue Rule: one durable queue for long tasks; state machine per job; retries + DLQ from day 1.
  • Happy-Path First: prove upload ‚Üí process ‚Üí search (or analogous) end-to-end before any embellishment.
  • Talk in Deltas: each evolution step should change exactly one constraint (throughput, latency, durability, cost).
  • Hot-spot Radar: say where hot keys and fan-outs will occur and how you‚Äôll detect them (metrics & sample alerts).
  • Observability trinity: logs with correlation IDs, RED/USE metrics, 1‚Äì2 SLOs with error budgets.

Reflection prompts (turn this sting into skill)

Answer these in writing‚Äîshort bullets:

  1. Priority drift: When under pressure, what made me reach for ‚Äúshiny‚Äù components before the core flow? How will I catch that impulse (e.g., say the 90-sec script before touching the marker)?
  2. Narrative gap: Could I explain the entire data path in 60 seconds without diagrams? If not, tighten it.
  3. State clarity: Where did I fail to name states and transitions? Rewrite them as a table.
  4. Numbers poverty: Which number did I avoid stating (traffic, size, cost)? Why? Add a default baseline toolkit (below).
  5. Failure courage: Which failure mode do I habitually ignore? Design one control and one alert for it.
  6. Life parallel: In your HRBP project, scope crept and you optimised mid-UAT. What‚Äôs the SD analogue? ‚Üí Lock MVP spine; create a ‚Äúparking lot‚Äù for nice-to-haves; only pull from it after the spine stands.

Baseline numbers toolkit (memorise reasonable defaults)

  • Payloads: JSON request 1‚Äì5 KB; doc text page 2‚Äì4 KB; image 200‚Äì500 KB.
  • Throughput tiers: prototype 10 rps; MVP 100 rps; scale target 1k rps.
  • Storage: plain text ~1 byte/char; embed vector 768 floats ‚âà 3 KB.
  • Queue: Pub/Sub/Kafka consumer stable at O(100‚Äì1k msg/s) per worker with I/O-bound tasks.
  • Latency: network hop 5‚Äì20 ms in-region; disk read 1‚Äì10 ms SSD; ANN/vector search 10‚Äì80 ms per 10k vectors (ballpark).
  • Cost: egress is often the silent killer; cache prevents egress, not just latency.

  1. ‚ÄúWalk the happy path, no optimisations.‚Äù
  2. ‚ÄúPoint to each arrow and tell me payload & size.‚Äù
  3. ‚ÄúWhat‚Äôs the shard key? Why won‚Äôt it hot-spot?‚Äù
  4. ‚ÄúIf this downstream is down for 10 minutes?‚Äù
  5. ‚ÄúHow do I retry safely without dup effects?‚Äù
  6. ‚ÄúWhich read hits which index?‚Äù
  7. ‚ÄúShow me the job state machine.‚Äù
  8. ‚ÄúWhere do you log correlation IDs?‚Äù
  9. ‚ÄúWhat‚Äôs your single biggest cost line?‚Äù
  10. ‚ÄúHow do you roll back a bad deploy?‚Äù

Practice protocol (repeat weekly)

  • 5-min warm-up: recite the 90-sec opening, draw the state machine.
  • 15-min drill: pick any common SD prompt (feed, rate limiter, search, chat). Do: numbers ‚Üí flow ‚Üí failures.
  • 10-min red-team: have a friend interrupt with the 10 questions above.
  • Post-mortem note (3 bullets): what I skipped, what I over-engineered, one number I‚Äôll bring next time.

Your cheat sheet, tuned (copy/paste)

System-Design Mistakes ‚Äî Pocket Checklist

  1. Vague spec ‚Üí Write goal, non-goals, SLOs.
  2. No numbers ‚Üí Quick calc: QPS, size, storage, latency, cost.
  3. Messy diagram ‚Üí Boxes, numbered arrows, legend; pass-1 < 2 min.
  4. Silent schema ‚Üí Entities, keys, indexes, hot-key risk.
  5. No state machine ‚Üí Jobs: PENDING‚ÜíRUNNING‚ÜíRETRYING‚ÜíSUCCEEDED|FAILED(DLQ).
  6. No data-flow story ‚Üí Trace one request with format/size/retry/idempotency.

  8. Zero trade-offs ‚Üí For each big choice, 2 pros/2 cons + reversal path.
  9. No roadmap ‚Üí MVP spine now; next 10√ó bottlenecks later.
